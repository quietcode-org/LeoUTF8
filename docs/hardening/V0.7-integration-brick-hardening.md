# LeoUTF8 V0.7 Integration Brick Hardening

## Status

This document defines the V0.7 hardening gate for LeoUTF8.

V0.7 is not a feature expansion release.

V0.7 is an integration-brick hardening phase.

## Cupertino 2009 Compliance

LeoUTF8 follows the Leopard-Crew system-first doctrine.

If Mac OS X 10.5.8, Apple documentation, Xcode 3.x, GCC, Cocoa,
CoreFoundation, Foundation, HFS+, or another native Leopard subsystem defines
a documented or system-typical rule for a software concern, LeoUTF8 follows
that rule before project-local style preferences.

See:

~~~text
docs/hardening/Cupertino-2009-compliance.md
~~~

## Core Rule

LeoUTF8 must reach a stable NS/CoreFoundation-level brick quality before it is
integrated into other Leopard-Crew projects beyond external probe use.

External consumer probes are allowed.

Interpreter-core patches, application-level integration, and public dependent
APIs are locked until V0.7 hardening is complete.

## What V0.7 Must Prove

LeoUTF8 V0.7 must prove that the brick is:

- buildable
- testable
- distributable
- externally consumable
- version-identifiable
- ownership-safe
- header-clean
- release-packable
- documented enough for another Leopard project to use without tribal knowledge

## Required V0.7 Work

### Versioning

LeoUTF8 must expose a clear version identity.

Required:

- version constants
- version string
- documented version policy

Candidate:

- `LeoUTF8Version.h`

### Header Hygiene

Public headers must be suitable for external consumers.

Required:

- stable include guards
- C++ compatibility where appropriate
- no dependency on private build paths
- no direct exposure of vendored utf8proc headers
- clear ownership comments

### Error Semantics

Status handling must be stable.

Required:

- documented `LeoUTF8Status`
- documented status messages
- predictable NULL-output cleanup
- invalid input behavior confirmed by probes

### Ownership Semantics

All returned memory and objects must have explicit ownership rules.

Required:

- `LeoUTF8Free` for Core buffers
- `CFRelease` for CoreFoundation Create/Copy results
- manual retain/release for Foundation results on Leopard

### Distribution

The existing `make dist` output must remain the official staged brick format.

Required layout:

dist/LeoUTF8/
  include/
    LeoUTF8.h
    LeoUTF8CoreFoundation.h
    LeoUTF8Foundation.h
  lib/
    libLeoUTF8Core.a
    libLeoUTF8CoreFoundation.a
    libLeoUTF8Foundation.a

### Release Artifact

V0.7 should add a reproducible local release artifact.

Candidate:

- `make release`
- `release/LeoUTF8-0.7-leopard-ppc.tar.gz`

The release archive must contain headers, libraries, docs, and examples.

### Examples

V0.7 should provide simple external examples separate from internal probes.

Candidate layout:

Examples/
  Core/
  CoreFoundation/
  Foundation/

Examples must consume LeoUTF8 through dist/release-style include and lib paths.

### Research Boundary

HFS+ research remains documented but not public API.

Locked:

- final HFS+ comparison API
- automatic filename normalization
- CFURL policy
- Finder policy
- APFS/SMB/AFP behavior

## Explicitly Allowed During V0.7

Allowed:

- hardening public headers
- adding version metadata
- improving Makefile targets
- adding release packaging
- adding examples
- strengthening documentation
- adding tests that verify existing behavior

## Explicitly Not Allowed During V0.7

Not allowed:

- LeooRexx interpreter-core integration
- Rexx string semantics changes
- LeoUTF8HFS public API
- HFS+ comparison implementation
- new Unicode feature expansion
- hidden normalization
- framework packaging unless static brick release is already stable

## Completion Criteria

V0.7 is complete when:

- `make clean`
- `make`
- `make check`
- `make dist`
- `make distcheck`
- `make hfscheck`
- `make hfsboundarycheck`
- `make hfscasecheck`
- `make hfscomparecheck`
- `make hfsurlcheck`

all run successfully on Leopard/PPC, and:

- version identity is public
- release artifact can be created locally
- examples build against the staged/release artifact
- documentation reflects the hardened brick state

## Doctrine

Bytes remain bytes.
UTF-8 is explicit.
Foundation and CoreFoundation are native Leopard text layers.
LeoUTF8Core guards the byte boundary.
LeoUTF8 does not silently normalize.
HFS+ behavior stays HFS+-scoped.
Other projects may probe LeoUTF8, but must not depend on unstable internals.
